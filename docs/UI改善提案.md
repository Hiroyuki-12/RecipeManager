Phase 5 完了時点（プロトタイプ忠実な React 化が終わった段階）で挙げた、操作性・見やすさに関する改善案。
機能追加ではなく既存 5 機能の UX を磨く方向の提案。Phase 6 以降または別 Issue で順次対応する想定。

インパクトの高い順に並べている。

---

## 高インパクト

### 1. フォームのバリデーションエラーが体感しづらい

**現状**: [frontend/components/RecipeForm.tsx](../frontend/components/RecipeForm.tsx) は submit 時にエラーメッセージをフィールド下部に表示するのみ。
- 入力欄自体の見た目が変わらない（赤枠などがない）
- エラー位置に自動スクロールされない
- 入力を直しても submit するまでエラーが消えない

**改善案**:
- エラー時、対象 input に `.has-error` クラスを付けて赤枠を表示
- onChange で該当エラーをリアルタイム解除
- submit 時に最初のエラーフィールドへ `scrollIntoView({ behavior: "smooth", block: "center" })`

### 2. 一覧カードの情報密度が低い

**現状**: タイトル + "カテゴリ / 調理時間" の 2 行のみ。フル幅 1 列で縦に間延び。

**改善案**:
- 2〜3 列のレスポンシブグリッド（CSS Grid `repeat(auto-fill, minmax(280px, 1fr))`）
- メタ情報を「🍴4人分 / ⏱30分 / 🥕材料5品」のようにアイコン+数値で並べる
- 更新日も小さく表示

### 3. 検索のデバウンス未実装

**現状**: [frontend/pages/recipes/index.tsx](../frontend/pages/recipes/index.tsx) は keyword の onChange で即 API を叩いている。1 文字打つたびにリクエストが飛ぶ。

**改善案**: 300ms のデバウンス。
```ts
useEffect(() => {
  const t = setTimeout(() => fetch(...), 300);
  return () => clearTimeout(t);
}, [keyword]);
```

## 中インパクト

### 4. 削除確認モーダルの危険操作の表現

**現状**: [frontend/pages/recipes/[id].tsx](../frontend/pages/recipes/[id].tsx) の確認モーダルで OK ボタンが目立つオレンジ（`btn-primary`）、キャンセルが地味。**破壊的操作なのに肯定ボタンが押しやすい**配色。

**改善案**:
- OK 側を `btn-danger`（赤）に
- キャンセル側を太字 or 強調枠に
- 文言を「削除する」と動詞化（誤クリック耐性向上）

### 5. 戻るリンク / アクションボタンの位置と階層感

**現状**: 詳細画面で「← 一覧へ戻る」が左上の地味なテキストリンク、対して「編集/削除」はタイトル右に大きく並ぶ。視線移動が非対称。

**改善案**: ヘッダー直下に固定位置のパンくず or 戻る矢印を統一配置。

### 6. 動的行追加の操作コスト

**現状**: 材料・手順とも `+ 追加` ボタンが破線下にぽつんとあり、行を増やすたびマウスが下に流れる。

**改善案**:
- 最終行で Enter キーを押したら次行を自動追加
- 最終行に値が入ったら自動で空行を 1 行追加（追従型 UX）

## 低インパクト

### 7. cooking_time / servings の入力欄が広すぎる

**現状**: `form-row` で 1/3 ずつの幅を取っていて、3 桁入力に対して間延び。

**改善案**: number 系は `flex: 0 0 120px` で固定、カテゴリだけ可変。

### 8. 詳細画面の見出し記号

**現状**: `【材料】` `【手順】` の墨括弧は和文資料感が強い。

**改善案**: `<h3>` で「材料」「作り方」とフラット化、または 🥕📝 アイコンで視認性向上（趣味の領域）。

### 9. ヘッダーから新規登録へ飛べない

**現状**: 詳細・登録・編集画面ではヘッダーロゴクリックでしか一覧に戻れず、新規登録にはまず一覧に戻る必要がある。

**改善案**: ヘッダー右側に常設の「+ 新規登録」ボタンを置く。
